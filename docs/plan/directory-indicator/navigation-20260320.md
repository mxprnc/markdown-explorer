# 디렉터리 내비게이션 및 권한 개선 계획 (2026-03-20)

이 문서는 Mark Explorer 애플리케이션의 "가짜" 경로 표시 문제를 해결하고 권한 요청 흐름을 간소화하기 위한 전략을 설명합니다.

## 1. 배경 및 문제점
현재 애플리케이션은 하드코딩된 `rootPath`(`~/Documents/toy-projects`)를 실제 경로 앞에 붙여서 표시하고 있습니다. 이는 사용자 파일 시스템의 실제 위치를 반영하지 않기 때문에 근본적으로 오해의 소지가 있습니다.

**제약 사항:**
- 웹 파일 시스템 액세스 API(Web File System Access API)는 보안상의 이유로 **절대 경로를 제공하지 않습니다.**
- 따라서 현재 표시되는 모든 절대 경로는 "추측"이거나 사용자가 수동으로 입력한 값입니다.

## 2. 제안하는 개선 사항

### A. "정직한" 내비게이션 헤더 (스마트 브레드크럼)
가짜 절대 경로를 보여주는 대신, 열려 있는 폴더 이름을 시작점으로 하는 브레드크럼을 표시합니다.
- **현재**: `~/Documents/toy-projects/4.project-강의필기-항로-LLM-Langraph`
- **개선안**: `📁 4.project-강의필기-항로-LLM-Langraph / lecture1 / lecturenote-1.md`
- **조치**: 경로 표시 로직에서 `rootPath` 의존성을 제거합니다.

### B. 사용자 정의 베이스 경로 (옵션 힌트)
사용자가 굳이 전체 절대 경로를 보고 싶은 경우, 루트를 "고정"할 수 있는 방법을 제공합니다.
- 폴더 이름 옆에 "베이스 경로 설정" 버튼/아이콘을 추가합니다.
- 사용자가 자신의 실제 베이스 경로(예: `/Users/alpha300uk/...`)를 붙여넣을 수 있게 합니다.
- 이를 `localStorage`에 저장하되, 실제 파일 시스템의 소스가 아닌 **'표시용 힌트'**로만 처리합니다.

### C. 선제적 권한 요청
현재 "로컬 폴더 열기" 흐름은 사용자가 저장을 시도할 때까지 기다리거나, 별도의 "⚠️ 수정권한 필요" 버튼을 클릭해야 합니다.
- **개선안**: `showDirectoryPicker` 호출 직후, 즉시 `handle.requestPermission({ mode: 'readwrite' })`를 실행합니다.
- 이를 통해 폴더 선택 제스처와 권한 요청을 하나로 묶어 사용자 편의성을 높입니다.

## 3. 구현 단계
1. **`app/index.tsx` 수정**:
   - `handleOpenDirectory`에서 폴더 선택 후 즉시 쓰기 권한을 요청하도록 변경합니다.
   - 헤더의 `headerTitle`과 푸터의 `footerPathText` 렌더링 로직을 폴더 상대 경로 중심으로 수정합니다.
2. **UI 다듬기**:
   - 익스플로러 헤더에서 현재 폴더 이름을 더 눈에 띄게 표시합니다.
   - 가독성을 위해 브레드크럼 컴포넌트를 개선합니다.

## 4. 검증 계획
- 폴더를 열 때 브라우저에서 즉시 "읽기/쓰기" 권한 요청 팝업이 뜨는지 확인합니다.
- "가짜" 루트 설정과 관계없이, 열린 폴더를 기준으로 경로가 정확하게 표시되는지 확인합니다.
