# 플러그인 생태계 구축 전략에 대한 피드백 및 조언 (2026-04-24)

사용자께서 제안하신 **GitHub Registry 기반의 플러그인 공유 방식**은 개인 개발자가 운영 비용 부담 없이 오픈소스 커뮤니티를 활성화하기에 매우 현실적이고 영리한 전략입니다. Obsidian이나 VSCode의 초기 생태계도 이와 유사한 형태(또는 이를 보완하는 형태)로 성장했습니다.

이 컨셉에 대해 기술적, 운영적 관점에서 몇 가지 조언과 구체적인 고려사항을 제안해 드립니다.

---

## 1. GitHub Registry 방식의 타당성 및 장점
- **비용 제로 (Cost-Effective)**: 별도의 서버 인프라(EC2, DB 등)가 필요 없으므로 유지비가 발생하지 않습니다.
- **투명성 (Transparency)**: 모든 플러그인 등록 과정이 PR(Pull Request)을 통해 이루어지므로, 코드 변경 이력이 공개되고 커뮤니티에 의한 검증이 가능합니다.
- **신뢰도 (Trust)**: 개발자가 자신의 GitHub 저장소에 소스 코드를 공개하고 배포하므로, 사용자 입장에서 악성 코드 여부를 확인하기 용이합니다.

---

## 2. 기술적 구현을 위한 구체적 조언

### ① 배포 및 다운로드 최적화 (CDN 활용)
GitHub API는 호출 제한(Rate Limit)이 있습니다. 앱에서 직접 GitHub API를 빈번하게 호출하는 대신, **JSDelivr**나 **GitHub Pages**를 활용하는 것이 좋습니다.
- **방법**: `plugins.json` 파일을 메인 레지스트리 저장소의 특정 브랜치에 두고, 이를 `https://cdn.jsdelivr.net/gh/user/repo@branch/plugins.json` 형태로 호출하면 API 제한 없이 캐싱된 데이터를 빠르게 가져올 수 있습니다.

### ② 보안 및 샌드박싱 (Security)
가장 어려운 숙제입니다. 타인이 작성한 JS 코드를 사용자의 로컬 환경에서 실행하는 것은 위험합니다.
- **최소 권한 원칙**: 플러그인이 파일 시스템 전체가 아닌, 특정 워크스페이스 디렉토리만 접근할 수 있도록 API를 제한해야 합니다.
- **Sandboxing**: 
    - **Logic Plugins**: Web Worker에서 실행하여 DOM 접근을 차단하고 메시지 통신으로만 코어와 대화하게 합니다.
    - **UI Plugins**: `iframe`의 `sandbox` 속성을 활용하여 권한을 제어할 수 있습니다.

### ③ 자동화된 검수 시스템 (CI/CD)
개인이 모든 PR을 수동으로 검수하는 것은 한계가 있습니다.
- **GitHub Actions 활용**: 
    - PR이 올라오면 `manifest.json`의 스키마가 올바른지 자동 체크합니다.
    - 등록된 다운로드 URL(ZIP 파일)이 유효한지 확인합니다.
    - 플러그인이 선언한 `minAppVersion`이 현재 앱과 호환되는지 검증합니다.

---

## 3. 단계적 발전 제안 (Hybrid 접근법)

Phase 2와 Phase 3 사이의 중간 단계를 고려해 볼 수 있습니다.

- **Phase 2.5 (Severless Metadata)**: 
    - 파일(ZIP)은 여전히 GitHub에 두되, 플러그인의 **조회수, 추천수(Like), 다운로드 수** 같은 동적인 데이터만 **Supabase**나 **Firebase**의 무료 티어를 사용하여 관리합니다. 
    - 이를 통해 서버 운영 비용을 최소화하면서도 마켓플레이스의 "인기순" 정렬 같은 소셜 기능을 추가할 수 있습니다.

---

## 4. 사용자 경험(UX) 측면의 조언

- **버전 호환성 알림**: 앱 버전이 업데이트되어 특정 플러그인이 동작하지 않을 때, 사용자에게 명확히 "호환되지 않는 플러그인"임을 알리고 비활성화하는 로직이 필수적입니다.
- **설치 편의성**: `plugins.json`에 GitHub Release의 다운로드 URL을 포함시켜, 앱 내에서 버튼 하나로 다운로드 -> 압축 해제 -> 플러그인 폴더 이동이 자동으로 이루어지게 설계하세요.

---

## 💡 결론
제안하신 **GitHub Registry 기반 방식은 훌륭한 시작점**입니다. 
백엔드 개발과 비용에 대한 부담을 내려놓고, 대신 **"플러그인 API의 견고함"**과 **"보안 샌드박싱"**에 더 집중하실 수 있다는 점에서 현재 상황에 가장 적합한 선택이라고 생각합니다. 

이러한 개방형 구조는 개발자들의 참여를 이끌어내는 데 큰 강점이 될 것입니다.
