# Phase 3: Automated Dev Server & Browser Open - Implementation Plan

## 🏗 시스템 구성
데스크톱 앱의 메인 프로세스(Node.js) 권한을 활용하여 하위 프로세스를 관리합니다.

### 1. 프로세스 런너 (Process Runner)
- `child_process.spawn`: 비동기적으로 `npm` 명령어를 실행하고 스트림 데이터를 수신.
- `TaskQueue`: 의존성 설치(`install`)와 서버 실행(`dev`)을 순차적으로 관리.

### 2. 브라우저 연동
- `shell.openExternal` (Electron): 운영체제의 기본 브라우저를 사용하여 특정 URL 오픈.
- `port-ready` 체크: `http-get` 또는 `net` 모듈을 사용하여 3000번 포트가 응답할 때까지 대기.

## 🛠 단계별 구현 계획

### Step 1: 환경 검사 (Pre-flight Check)
- [ ] 사용자 시스템에 `node` 및 `npm`이 설치되어 있는지 확인하는 유틸리티 구현.
- [ ] 미설치 시 내보내기 마법사에서 경고 노출 및 자동 실행 옵션 비활성화.

### Step 2: 프로세스 제어 모듈 개발
- [ ] `ProcessManager` 클래스 구현: `start()`, `stop()`, `on('data', ...)` 등의 인터페이스 제공.
- [ ] 실행 중인 서버를 강제 종료할 수 있는 Kill 로직 포함.

### Step 3: UI 로그 뷰어 통합
- [ ] 내보내기 완료 화면에 터미널 형태의 로그 콘솔 컴포넌트 추가.
- [ ] 서버 상태(준비 중, 설치 중, 실행 중, 에러)를 시각적인 인디케이터로 표현.

### Step 4: 브라우저 자동 오픈 로직
- [ ] 서버 로그에서 "Ready in..." 메시지를 감지하거나 포트 모니터링을 통해 브라우저 오픈 타이밍 결정.
- [ ] 자동으로 `http://localhost:3000` 호출.

## 🧪 테스트 계획
- **서버 구동 테스트**: 다양한 운영체제(Windows, macOS) 환경에서 `npm` 명령어가 정상적으로 전달되는지 확인.
- **프로세스 고립 테스트**: Mark Explorer 앱을 종료했을 때 생성된 개발 서버 프로세스가 함께 소멸되는지 확인 (좀비 프로세스 방지).
- **에러 복구 테스트**: `npm install` 중 네트워크 오류 발생 시 재시도 또는 에러 메시지가 정확히 노출되는지 확인.
