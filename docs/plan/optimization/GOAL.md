# 성능 최적화 목표 및 실행 계획

## 1. 현황 분석 및 기법 적용 상태
현재 Markdown Explorer 프로젝트에 적용된 주요 최적화 기법들의 현황입니다. 분석 결과, 이미 상당 부분 현대적인 에디터 최적화 기법들이 반영되어 있습니다.

| 기법 | 적용 상태 | 상세 내용 |
| :--- | :---: | :--- |
| **가상 스크롤링** | 부분 적용 | 에디터(Tiptap)는 전체 DOM을 그리지만, CodeMirror 모드 등에서 부분적으로 효율적인 렌더링을 시도함. |
| **점진적 파싱** | **기적용** | Tiptap의 기반인 ProseMirror를 통해 변경된 노드만 AST 단위로 업데이트함. |
| **웹 워커 (Web Workers)** | **미적용** | 문서 파싱 및 무거운 연산(Math, Mermaid 등)이 메인 스레드에서 동작 중. |
| **DOM 업데이트 최적화** | **기적용** | React의 Virtual DOM Diffing 및 에디터 콘텐츠 업데이트 시 Debounce(300ms) 적용 완료. |
| **실시간 하이브리드 모드** | **기적용** | `LiveMarkdownExtension`을 통해 커서 위치에 따른 마크다운 기호 동적 노출(Typora 방식) 구현 완료. |

---

## 2. 향후 개선 목표 (Optimization Goals)

문서의 크기가 커지거나 복잡한 수식/다이어그램이 많아질 때도 최상의 사용자 경험을 유지하기 위해 다음 두 가지를 핵심 목표로 설정합니다.

### 🎯 목표 1: Web Worker를 이용한 파싱 분리
Markdown을 HTML로 변환하는 파싱 프로세스와 복잡한 렌더링 연산을 메인 UI 스레드에서 분리합니다.
*   **기대 효과:** 대용량 문서 편집 시 타이핑 지연(Input Lag)을 완전히 제거하고 UI 반응성을 극대화함.
*   **실행 내용:** `remark`, `rehype` 파싱 로직을 별도의 워커 파일로 분리하고 비동기 통신 구조로 변경.

### 🎯 목표 2: 미리보기(Preview) 영역 메모이제이션 고도화
현재 전체 문서를 다시 파싱하고 렌더링하는 미리보기 영역을 최적화합니다.
*   **기대 효과:** 특정 섹션 수정 시 해당 부분만 다시 렌더링하여 CPU 사용율을 개선하고 배터리 소모를 줄임.
*   **실행 내용:** 
    *   문서를 블록(Block) 단위로 분할하여 관리.
    *   `React.memo` 및 고유 ID를 활용해 변경되지 않은 블록의 리렌더링을 건너뜀.
    *   AST 수준에서의 Diffing을 통한 부분 업데이트 로직 검토.
