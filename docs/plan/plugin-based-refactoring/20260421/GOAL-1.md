# 플러그인 아키텍처 기반 인프라 구축 목표 (GOAL-1.md)

옵시디언(Obsidian)과 같은 확장 가능한 생태계를 구축하기 위해, Mark Explorer의 핵심 기능을 플러그인 형태로 분리하고 외부 기능을 동적으로 수용할 수 있는 기반 시스템을 설계 및 구현합니다.

## 🎯 최종 목표
- [ ] **플러그인 규약 정의**: 모든 플러그인이 따라야 할 표준 인터페이스(Base Class)가 수립되었는가?
- [ ] **중앙 집중식 관리**: 플러그인의 생명주기(Load/Enable/Disable/Unload)를 제어하는 `PluginManager`가 존재하는가?
- [ ] **애플리케이션 API 노출**: 플러그인이 앱의 상태(Workspace, FileSystem)에 접근할 수 있는 안전한 `App` 객체가 제공되는가?
- [ ] **기능 모듈화**: 기존의 독립적 기능(예: Gemini Chat)을 플러그인 구조로 전환할 준비가 되었는가?
- [ ] **MCP(Model Context Protocol) 지원**: 플러그인이 MCP 서버를 등록하고, AI 모델이 해당 도구(Tools)를 사용할 수 있는 환경이 구축되었는가?

## ✅ 성공 기준
1. **확장성**: 새로운 기능을 추가할 때 코어 엔진(`app/` 폴더 내 코드)을 수정하지 않고 독립적인 모듈 추가만으로 가능해야 함.
2. **MCP 호환성**: 외부 MCP 서버를 플러그인을 통해 연결하고, 앱 내 AI가 그 도구들을 실시간으로 인식하여 호출할 수 있어야 함.
3. **안정성**: 특정 플러그인에서 오류가 발생해도 전체 애플리케이션이 크래시되지 않도록 격리 및 예외 처리가 되어야 함.
4. **유연성**: 사용자가 런타임에 특정 기능을 실시간으로 켜고 끌 수 있어야 함 (Hot-swapping).
5. **일관성**: 모든 확장 포인트(사이드바, 명령어 팔레트 등)가 동일한 API를 통해 일관된 방식으로 제공되어야 함.
