# JavaScript Performance Issues in Large Files

파일이 길어질수록 JS가 느려지는 이유는 단순히 "글자 수가 많아서"라기보다, **브라우저(또는 런타임)가 그 코드를 해석하고 실행하는 방식** 때문입니다. 크게 세 가지 관점에서 볼 수 있습니다.

---

### 1. 구문 분석(Parsing)과 컴파일 비용
JS는 코드를 실행하기 전에 엔진(V8 등)이 전체 코드를 읽어 **AST(Abstract Syntax Tree, 추상 구문 트리)**라는 구조로 변환해야 합니다.

* **비대해진 트리:** 파일이 길어지면 이 트리 구조가 기하급수적으로 복잡해집니다. 엔진은 변수 선언, 함수 스코프 등을 하나하나 확인하며 지도를 그리는데, 파일이 길면 이 단계에서 CPU 점유율이 치솟고 시간이 오래 걸립니다.
* **JIT 컴파일:** JS는 실행 직전에 기계어로 바꾸는 JIT(Just-In-Time) 컴파일러를 쓰는데, 코드가 방대하면 최적화해야 할 대상이 너무 많아져서 실행 초기 속도가 뚝 떨어집니다.



### 2. 단일 스레드(Single Thread)의 한계
JavaScript는 기본적으로 **한 번에 한 가지 일만 하는 싱글 스레드** 방식입니다.

* **메인 스레드 점유:** 아주 긴 JS 파일을 로드하고 해석하는 동안 브라우저의 메인 스레드는 그 작업에 매달립니다. 이 기간 동안 사용자가 버튼을 클릭하거나 화면을 스크롤해도 브라우저는 응답하지 못하고 "버벅임(Jank)"이 발생하게 됩니다.
* **가비지 컬렉션(GC) 부담:** 코드량이 많고 다루는 데이터가 클수록, 더 이상 안 쓰는 메모리를 찾아 지우는 '가비지 컬렉션' 작업이 더 자주, 더 오래 실행됩니다. 이게 실행 중간중간 흐름을 끊어먹죠.

### 3. 스코프 체인(Scope Chain) 탐색 성능
JS는 변수를 찾을 때 현재 위치에서 시작해 위로 올라가며 찾는 '스코프 체인' 방식을 씁니다.

* 파일이 수만 줄에 달하고 함수가 겹겹이 쌓여 있으면, 특정 변수 하나를 찾기 위해 거쳐야 할 단계가 많아집니다.
* 특히 전역 변수를 남발하거나 클로저(Closure)가 복잡하게 얽힌 긴 파일일수록 이 탐색 과정이 성능 저하의 원인이 됩니다.

---

### 💡 해결책: 왜 "모듈화"를 강조할까?
그래서 요즘은 하나의 거대한 파일(`index.js` 1만 줄) 대신, **작은 모듈(수백 줄 단위의 여러 파일)**로 쪼개는 방식을 씁니다.

1. **Lazy Loading:** 당장 필요한 부분만 먼저 읽고, 나머지는 나중에 읽어서 초기 로딩 속도를 높입니다.
2. **병렬 처리:** 브라우저가 여러 파일을 동시에 다운로드하고 필요한 시점에만 해석하게 유도합니다.
3. **트리 쉐이킹(Tree Shaking):** 빌드 도구(Webpack, Vite 등)가 긴 파일 안에서 실제로 안 쓰는 코드를 찾아내서 삭제해 버립니다.

### 🧐 Q. 서버(K8s, Docker)의 리소스(CPU/RAM)를 높여서 성능을 개선할 수 있을까요?

**정답은 "아니오"에 가깝습니다.**

* **서버 리소스:** 주로 웹 서버가 정적 파일(HTML, JS, CSS)을 사용자에게 내려주는 속도나, 동시에 접속할 수 있는 사용자 수에 영향을 줍니다.
* **마크다운 렌더링 성능:** 이 프로젝트(Markdown Explorer)는 대부분의 연산이 사용자의 **브라우저(Client-side)**에서 일어납니다. 따라서 파일이 길어서 발생하는 렉(Lag)은 서버 사양보다는 사용자의 PC 성능과 **스크립트 최적화 방식**이 더 중요합니다.

즉, 서버 사양을 높이는 것보다 **[개선 방향 제안서](./suggestion-why-slow-large-file.md)**에 적힌 대로 클라이언트 측 로직을 최적화하는 것이 근본적인 해결책입니다.
