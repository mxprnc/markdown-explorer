# 대용량 파일 성능 개선 방향 및 실행 계획

`why-slow-large-file.md`에서 분석한 원인(구문 분석 비용, 싱글 스레드 점유, 스코프 탐색 부하)을 바탕으로, Mark Explorer 프로젝트를 위한 현실적인 개선 계획을 제안합니다.

---

## 1. 핵심 개선 방향

### 🛠 코드 분할 (Code Splitting & Modularization)
* **목표:** 하나의 거대한 파일(예: `app/index.tsx`, 2600+줄)을 기능 단위의 작은 모듈로 쪼갭니다.
* **효과:** 브라우저가 한 번에 해석해야 할 AST 크기를 줄이고, JIT 컴파일러의 부담을 분산시킵니다.

### 💨 비동기 및 지연 로딩 (Lazy Loading)
* **목표:** 당장 화면에 보이지 않는 기능(예: 설정 모달, 특정 필터 로직)은 필요할 때만 불러옵니다.
* **효과:** 앱 초기 구동 속도와 메모리 점유율을 최적화합니다.

### 🧵 메인 스레드 해방 (Offloading Main Thread)
* **목표:** 텍스트 분석, 복잡한 마크다운 파싱 등 연산 집약적인 작업은 Web Worker를 활용하거나 마이크로 태스크로 쪼갭니다.
* **효과:** UI 렌더링과 사용자 입력 반응성을 유지(Jank 방지)합니다.

---

## 2. 단계별 실행 계획

### [1단계] 컴포넌트 추출 (Immediate)
현재 `app/index.tsx`에 몰려 있는 UI 로직을 독립적인 컴포넌트로 분리합니다.
* **Header 컴포넌트:** 경로 표시, 테마 토글, 폴더 열기 로직 분리.
* **Sidebar 컴포넌트:** 파일 시스템 트리 브라우저 로직 분리.
* **Modal 컴포넌트:** Gemini 설정, 이름 변경 등 모달을 각각 별도 파일로 분리.

### [2단계] 비즈니스 로직 분리 (Hook Pattern)
상태 관리와 파일 시스템 엔진 로직을 Custom Hooks로 추출합니다.
* `useFileSystem.ts`: 파일 읽기/쓰기, 디렉터리 스캔 로직.
* `useEditorState.ts`: 에디터 텍스트, 탭 관리, 저장 로직.
* `useGemini.ts`: AI 채팅 및 API 연동 로직.

### [3단계] 렌더링 최적화 (Advanced)
데이터가 많아질 경우를 대비한 렌더링 기법을 도입합니다.
* **Windowing/Virtualization:** 파일 탐색기에 항목이 수천 개일 경우, 화면에 보이는 부분만 렌더링하는 `react-window` 등의 기법 적용.
* **Incremental Parsing:** 아주 긴 마크다운 문서는 한 번에 전체를 파싱하지 않고 구역별로 나누어 처리하도록 구성.

---

## 3. 기대 효과
1. **유지보수성 향상:** 수천 줄의 코드를 뒤질 필요 없이 명확한 파일 구조에서 작업 가능.
2. **반응 속도 개선:** 파일 로드 및 초기 렌더링 시간이 단축됨.
3. **안정성:** 특정 모듈의 오류가 전체 앱의 중단으로 이어질 가능성 감소.
