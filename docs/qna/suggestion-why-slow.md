### 현재 프로젝트(markdown-explorer-v1) 성능 개선을 위한 현실적인 제안

`Tiptap` 기반의 에디터(`Editor.web.tsx`)와 `React-Markdown` 기반의 프리뷰(`Preview.web.tsx`)의 현재 구조를 바탕으로 바로 적용 가능한 최적화 방안입니다.

#### 1. 에디터: 선택 영역 업데이트 최적화 (Memoization)
현재 코드 블록, 제목 등 모든 커스텀 노드 뷰 가 `editor.on('selectionUpdate', ...)`를 각각 구독하고 있습니다.
*   **문제점**: 문서에 수백 개의 제목이나 코드 블록이 있을 경우, 커서가 한 칸 움직일 때마다 모든 컴포넌트가 동시에 리렌더링을 시도합니다.
*   **해결책**: 
    - `React.memo`를 사용하여 `node.textContent`나 `isEditing` 상태가 실제로 변하지 않았다면 리렌더링을 방지합니다.
    - 각 노드가 직접 구독하는 대신, 더 효율적인 전역 상태 관리 방식을 고려합니다.

#### 2. 프리뷰: 디바운싱(Debouncing) 적용
에디터에서 글을 입력할 때마다 프리뷰가 즉시 다시 파싱되고 렌더링되고 있습니다.
*   **문제점**: 수천 줄의 긴 문서에서 매 타자마다 전체 마크다운을 다시 그리는 것은 CPU에 큰 부담을 줍니다.
*   **해결책**:
    - [`app/index.tsx`](file:///Users/alpha300uk/workspace/alpha300uk/0.toy-project/markdown-explorer/project/markdown-explorer-v1/app/index.tsx)에서 `Preview`에 전달할 데이터를 별도의 `deferredContent` 상태로 관리합니다.
    - 사용자가 타이핑을 멈춘 뒤 약 200~300ms 후에만 프리뷰 데이터를 업데이트하여 렌더링 횟수를 획기적으로 줄립니다.

#### 3. 외부 라이브러리 초기화 중복 제거 (Mermaid)
여러 컴포넌트 내부에서 `mermaid.initialize`가 반복 호출되고 있습니다.
*   **문제점**: 다이어그램이 여러 개일 경우 매번 불필요한 초기화 리소스가 소모됩니다.
*   **해결책**:
    - `mermaid.initialize`를 컴포넌트 외부나 `App` 초기화 단계에서 **딱 한 번만** 실행되도록 변경하고, 개별 컴포넌트는 렌더링만 담당하게 합니다.

#### 4. 코드 하이라이터 스타일 객체 고정
`SyntaxHighlighter`의 스타일이 리렌더링마다 새로 생성될 수 있습니다.
*   **해결책**: 테마에 따른 스타일 객체를 변수로 고정하여 불필요한 객체 생성을 막습니다.

#### 5. 가상화 (Virtualization) 단계적 도입
문서가 극한으로 길어질 경우(수만 줄 이상), 사실상 브라우저의 DOM 개수 자체가 문제가 됩니다.
*   **현실적인 접근**: 당장 전체 구조를 바꾸기 어렵다면, 프리뷰 영역에서 현재 스크롤 위치에 해당하지 않는 섹션은 렌더링을 생략하는 방식을 단계적으로 검토합니다.
