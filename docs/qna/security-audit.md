# Gemini CLI를 활용한 보안 충족성 검사 Q&A (security-audit.md)

## Q: CI/CD 과정에서 보안 충족성 검사를 어떻게 수행하는 것이 좋을까요?

**A:** 프로젝트의 특성(로컬 파일 시스템, 플러그인 아키텍처, AI 연동)을 고려하여 4가지 핵심 영역에 대한 검사가 필요합니다.
1. **의존성 취약점(SCA)**: `npm audit` 등을 통한 라이브러리 보안 확인.
2. **비밀정보 유출(Secret Scanning)**: API 키 커밋 방지.
3. **정적 분석(SAST)**: ESLint 등을 통한 보안 패턴 검사.
4. **플러그인 샌드박싱**: 플러그인의 과도한 권한 사용 및 자원 해제 여부 검증.

---

## Q: Gemini CLI를 이용한 보안 검사 방식이 더 나은 방식이 될 수 있을까요?

**A:** 네, 매우 강력한 보완책이 될 수 있습니다. 전통적인 도구와 Gemini CLI(AI 기반)는 서로 상호보완적입니다.

### 1. 비교 분석
| 구분 | 전통적 도구 (ESLint, npm audit) | Gemini CLI (AI 기반) |
| :--- | :--- | :--- |
| **방식** | 알려진 취약점 DB, 정기 규칙 | **코드 문맥 및 로직 의도 분석** |
| **강점** | 속도, 확정적 결과 | **논리 결함, 설계 취약점 탐지** |
| **특징** | '실수'를 잡아냄 | **'취약한 설계'를 잡아냄** |

### 2. 이 프로젝트에서의 장점
- **플러그인 검증**: 플러그인이 `App` API를 우회하여 시스템 자원에 접근하려는지 문맥적으로 파악 가능.
- **맞춤형 리뷰**: `coding-style.md`에 정의된 특수 규칙(예: `onunload` 강제)을 준수하는지 AI가 정밀 검토.
- **MCP 보안**: 도구 호출 시나리오에서 발생할 수 있는 데이터 유출 위험 예측.

### 3. 결론: 하이브리드 접근 방식 (Hybrid Approach)
가장 좋은 방법은 **전통적 도구로 '알려진 문제'를 빠르게 걸러내고, Gemini CLI로 '논리적 결함'을 심층 분석**하는 하이브리드 워크플로우를 구축하는 것입니다.

## Q: 향후 구현 계획은 어떻게 되나요?

**A:** `GOAL-4.md`와 `PLAN-4.md`에 따라 다음을 진행합니다.
1. Gemini API를 활용한 보안 리뷰 스크립트(`scripts/security-audit.ts`) 개발.
2. 변경된 코드(Git Diff)만 선별하여 AI에게 검사를 요청하는 자동화 구축.
3. GitHub Actions와 연동하여 PR 제출 시 보안 리포트 자동 생성.
