## Q1
typora 나 stackedit 은 마크다운 내용이 길어질때 상대적으로 속도 저하가 덜한것 같은데 어떤 성능 최적화나 기법들이 적용되어 있을까?

## A1
마크다운 에디터에서 문서가 길어질 때도 성능을 유지하는 비결은 결국 **"화면에 보이지 않는 것은 그리지 않는다"**와 **"바뀐 부분만 수정한다"**는 원칙에 있습니다. Typora나 StackEdit 같은 현대적인 에디터들이 사용하는 주요 기법들을 정리해 드릴게요.

---

### 1. 가상 스크롤링 (Virtual Scrolling / Windowing)
가장 핵심적인 기술입니다. 수만 줄의 텍스트가 있어도 브라우저는 수만 개의 DOM 요소를 한꺼번에 렌더링하지 않습니다.
* **원리:** 현재 사용자의 **뷰포트(화면)** 내에 보이는 부분과 그 위아래 아주 일부만 실제 DOM으로 생성합니다.
* **효과:** 스크롤을 내릴 때마다 기존 요소를 재사용하거나 빠르게 교체하기 때문에, 문서 전체 길이에 상관없이 메모리 점유율을 일정하게 유지합니다.

### 2. 점진적 파싱 및 렌더링 (Incremental Parsing)
문서 전체를 매번 처음부터 끝까지 다시 해석하는 것은 비효율적입니다.
* **원리:** 사용자가 한 줄을 수정하면 에디터는 **추상 구문 트리(AST)**를 활용해 수정된 노드와 그에 영향을 받는 최소한의 영역만 다시 계산합니다.
* **효과:** 10,000줄의 문서에서 한 단어를 고칠 때 10,000줄을 다 다시 그리는 게 아니라, 해당 문단 혹은 블록만 업데이트하여 CPU 부하를 줄입니다.

### 3. 웹 워커 (Web Workers) 활용
마크다운을 HTML로 변환하는 파싱 작업은 연산 집약적인 작업입니다.
* **원리:** 메인 스레드(UI 스레드)가 화면을 그리고 사용자의 입력을 받는 동안, **Web Worker**라는 별도의 백그라운드 스레드에서 마크다운 파싱을 처리합니다.
* **효과:** 무거운 문서라도 타이핑 시 화면이 버벅거리거나 입력 지연(Input Lag)이 발생하는 현상을 방지합니다.

### 4. DOM 업데이트 최적화 (Diffing 알고리즘)
StackEdit 같은 브라우저 기반 에디터는 React나 Vue와 유사한 **Virtual DOM** 개념을 활용합니다.
* **원리:** 이전 상태의 렌더링 결과와 현재 상태를 비교(Diffing)하여, 실제로 변경된 HTML 요소만 교체합니다.
* **효과:** 브라우저의 리플로우(Reflow)와 리페인트(Repaint) 횟수를 획기적으로 줄여 렌더링 속도를 높입니다.

### 5. Typora만의 특수 기법: 실시간 하이브리드 모드
Typora는 편집 창과 미리보기 창이 분리되지 않은 **WYSIWYG(What You See Is What You Get)** 방식을 사용합니다.
* **원리:** 사용자가 포커스를 둔 요소만 마크다운 소스 코드로 보여주고, 포커스를 벗어나면 즉시 렌더링된 결과로 바꿉니다. 
* **효과:** 두 개의 창을 동시에 띄워 동기화(Sync Scroll)할 필요가 없으므로 리소스를 절반 수준으로 아낄 수 있습니다.

---

### 요약하자면
이 에디터들은 **"전체 데이터를 가지고는 있되, 브라우저가 처리해야 할 DOM의 양은 항상 최소한으로 유지"**하는 전략을 취하고 있습니다. 

개발자 입장에서 보자면, 대규모 데이터를 다룰 때 `Canvas`를 직접 쓰거나 `Virtual List` 라이브러리를 사용하는 것과 비슷한 원리가 텍스트 에디터 엔진(Codemirror, Monaco Editor 등) 수준에서 구현되어 있다고 보시면 됩니다.
