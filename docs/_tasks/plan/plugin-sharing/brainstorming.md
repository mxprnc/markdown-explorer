# 플러그인 공유 및 마켓플레이스 브레인스토밍 (brainstorming.md)

Mark Explorer의 생태계 확장을 위한 플러그인 공유 시스템 및 마켓플레이스 구축에 대한 아이디어 정리 문서입니다.

---

## 🎯 비전 및 목표
- **사용자 경험 극대화**: 사용자가 필요한 기능을 직접 찾아 쉽게 설치하고 관리할 수 있는 환경 제공.
- **커뮤니티 중심 성장**: 개발자들이 자신만의 플러그인을 공유하고 기여할 수 있는 플랫폼 구축.
- **플랫폼화**: 단순한 에디터를 넘어 다양한 도구가 결합되는 마크다운 에코시스템 구축.

## 🚀 단계별 구현 전략

### Phase 1: 로컬 플러그인 기반 다지기 (현재)
- 플러그인 API의 안정성 확보.
- `manifest.json` 표준 규격 정의 (ID, Name, Version, Author, Description 등).
- 플러그인 생명주기(`onload`, `onunload`) 관리 로직 완성.

### Phase 2: 가벼운 공유 시스템 (GitHub Registry 방식)
- **방식**: 별도의 백엔드 없이 GitHub Repository를 레지스트리로 활용.
- **특징**:
    - `plugins.json`에 플러그인 목록을 관리하고 PR을 통해 등록.
    - 앱 내에서 GitHub API를 통해 최신 플러그인 목록을 가져오고 ZIP 다운로드.
- **장점**: 서버 운영 비용 0, 커뮤니티 투명성 확보, 초기 도입 용이.

### Phase 3: 고도화된 플러그인 마켓 (Custom Backend 방식)
- **방식**: 전용 API 서버 및 데이터베이스 구축.
- **주요 기능**:
    - 사용자 인증(Auth) 및 개발자 대시보드.
    - 플러그인 업로드, 버전 관리, 통계 확인.
    - 사용자 리뷰 및 평점 시스템.
- **고려사항**: 서버 유지 비용, 데이터 보안, 운영 정책 수립 필요.

## 🛠 아키텍처 설계 시 고려사항 (Marketplace-Ready)

1. **플러그인 매니페스트 표준화**:
   - 나중에 마켓 서버가 메타데이터를 쉽게 파싱할 수 있도록 정교한 JSON 규격 설계.
2. **버전 호환성 체크**:
   - 코어 앱 버전과 플러그인 버전 간의 의존성(`minAppVersion`) 필드 도입.
3. **App API 추상화 및 캡슐화**:
   - 플러그인이 코어 내부 로직에 직접 접근하지 못하도록 전용 `App` 인터페이스 객체를 통해서만 자원을 사용하게 강제.
4. **보안 및 샌드박싱 (Sandboxing)**:
   - 외부 개발자가 만든 코드를 실행할 때 로컬 파일 시스템의 안전을 보장하기 위한 보안 계층(예: Web Worker 사용 또는 권한 제한) 검토.

## ⚠️ 향후 과제
- 플러그인 개발자 가이드 및 문서화.
- 공식 플러그인과 커뮤니티 플러그인의 구분 정책.
- 악성 플러그인 방지를 위한 검수 시스템 또는 보안 모델 수립.
